## 3. 参数命名的注意点

Lua-like 导出文本中函数头是：

```lua
function f_0000243B(a0, a1, a2, a3, a4, a5)
```

但函数体里实际又访问了 `a6`。这与 `f_00002403` 一样，并不是说底层函数真有 7 个显式参数，而是当前 IR 导出器对负偏移 `push_stack` 的参数名存在整体右移 1 的现象。

本函数涉及的取参指令如下：

- `10 F9` = `push_stack -7`
- `10 FA` = `push_stack -6`
- `10 FB` = `push_stack -5`
- `10 FC` = `push_stack -4`
- `10 FD` = `push_stack -3`
- `10 FE` = `push_stack -2`

在导出的 Lua-like 文本里，它们分别显示成：

- `a1`
- `a2`
- `a3`
- `a4`
- `a5`
- `a6`

因此，更合理的理解是：

> 本函数实际消耗的是 **6 个传入值**，只是导出文本把参数名整体右移了 1 位。

后文的伪代码会把它们重新记作 `p1..p6`。

### 8.2 结构共性

这类函数基本都呈现出同一个骨架：

```lua
function f_xxxxxxxx(p1, p2, p3, p4, p5, p6)
    G[some_flag] = true
    f_00010271()
    f_00051420(resource_name, p1, 1, nil, p2, p3, p4, p5, nil, nil)
    f_00010298(nil, nil, p6)
    return
end
```

也就是说，其共性非常稳定：

1. **先写一个专属全局位**：通常像“已看过 / 已进入 / 当前启用”的资源标记
2. **统一前处理**：总是通过 `f_00010271()` 进入显示前初始化
3. **固定模板调用显示核心**：真正变化的只有资源字符串与标记位
4. **统一后处理**：末尾总是把第 6 个实参交给 `f_00010298()`
5. **没有复杂控制流**：多数是单基本块、无条件直通返回

### 8.3 资源命名共性

资源名大多符合：

```text
<角色或分组前缀>_e<事件号><差分字母><差分编号>
```

例如：

- `KURO_e011a1`
- `KURO_e011b1`
- `HARU_e021d3`
- `CHIWA_e104e2`
- `ETC_e006g2`

从命名上看：

- 前缀对应角色或资源组
- `eNNN` 很像事件 / 章节 / CG 组编号
- 后缀 `a/b/c/...` 与 `1/2/3/...` 对应同一事件图的差分或分镜版本

### 8.4 参数角色的推测共性

基于模板与下层函数行为，可以对 6 个输入槽做相对保守的推测：

- `p1`：主显示控制参数，直接进入 `f_00051420()` 的第二参数位
- `p2 ~ p5`：显示附加参数组，可能对应位置、模式、过渡、附加控制或默认值覆盖
- `p6`：不参与主显示调用，只进入 `f_00010298()`，更像显示后的等待 / 收尾 / 模式控制参数

目前最稳妥的结论不是给它们下死语义，而是：

> 这 6 个参数中，前 5 个服务于“主显示入口”，第 6 个服务于“显示后处理入口”。

- 标记对应资源状态
- 进入公共初始化
- 把资源名和参数模板送进公共显示核心
