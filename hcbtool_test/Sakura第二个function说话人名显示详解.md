# Sakura 第二个类似 function（`f_00001918`）说话人名显示详解

## 0. 解析对象

本文解析 `Sakura.lua` 中用户给出的第二个“说话人名显示函数”：

```lua
function f_00001918(a0, a1, a2, a3, a4)
```

用户提供的信息表明，这一段与前面解析过的首个函数同属“文本框左侧说话人名显示控制”系统。

结合原始 HCB 字节串和 Lua-like IR，可以确认这段逻辑主要控制的是：

- **大雅** 的正常显示
- **ぼく** 的特殊显示
- **？？？ / ？大雅** 的未公开显示

因此它不是单纯的“显示大雅名字”，而更像是：

> **主角/大雅名字栏显示函数**

为了保持语义中立，本文暂命名为：

```text
ShowSpeakerName_TaigaLike(...)
```

或更细一点：

```text
ShowSpeakerName_大雅_ぼく_？？？(...)
```

## 4. 关键子函数关系

这段函数仍然调用了和前一个类似函数相同的关键子函数，但参数和语义略有不同。

---

## 4.1 `f_000019E0(99)`：名字栏样式初始化

调用：

```lua
S0 = 99
G[227] = S0
S0 = G[227]
__ret = f_000019E0(S0)
```

这里和前一个函数的最大不同是：

- 前一个函数设置的是 `G[227] = 1`
- 当前函数设置的是 `G[227] = 99`

而 `f_000019E0` 中确实存在对 `99` 的分支判断：

```lua
S0 = a1
S1 = 99
S0 = (S0 == S1)
```

并在对应分支中调用：

- `TextOutSize`
- `TextColor`
- 内部辅助函数 `f_0008CC08`

### 这说明什么？

这说明：

> `G[227]` 不能简单理解成“纯角色编号”，它更像是“当前名字栏样式/角色显示类别编号”。

对普通角色，`G[227]` 可能直接等于角色编号；
但对主角/特殊说话人，它可能使用一个**专门的样式编号 99**。

因此这里应更准确地说：

```text
G[227] = 当前说话人名字栏样式 ID
```

而不只是“当前角色 ID”。

---

## 4.2 `f_0004CCDA(name, alt_name)`：设置并打印名字栏文本

这部分与前一个函数相同，调用模式如下：

### 未公开分支
```lua
f_0004CCDA("　 ？？？ 　", "　 ？大雅 　")
```

### 特殊“ぼく”分支
```lua
f_0004CCDA("　　ぼく　　", nil)
```

### 正常“大雅”分支
```lua
f_0004CCDA("　　大雅　　", nil)
```

其内部关键逻辑还是：

- `G[233] = 当前要显示的名字`
- `G[234] = 备用名/真实名`
- `G[235] = 最近一次非空有效名字`
- `TextClear(8)` 清空名字栏槽
- `TextPrint(8, G[233])` 打印名字栏文本

所以：

> 这依旧是名字栏系统真正执行“清空并重绘左侧说话人名”的核心函数。

---

## 4.3 `f_000977B8(0, true)`：同步说话人显示状态

当前函数三次都是：

```lua
S0 = 0
S1 = true
__ret = f_000977B8(S0, S1)
```

与前一个函数不同：

- 前一个函数传的是 `G[227]`（即 1）
- 当前函数传的是固定 `0`

### 含义推测

这说明这里存在两套不同的“编号域”：

| 编号 | 可能意义 |
|---|---|
| `99` | 名字栏样式/特殊显示类别编号 |
| `0` | 主角/大雅 对应的角色逻辑编号 |

也就是说：

- 样式初始化走 `99`
- 后续角色状态同步走 `0`

这与主角类人物经常有特殊显示规则的情况非常一致。

因此这里可以推断：

> 当前函数更像是“主角/大雅名字显示函数”，只是其显示样式与普通角色不同，所以用 `99` 选样式，用 `0` 走角色同步与属性查表。

---

## 4.4 `f_0008D409(0)`：取得角色 0 的属性并缓存

末尾：

```lua
S0 = 0
__ret = f_0008D409(S0)
S0 = __ret
G[295] = S0
```

而 `f_0008D409` 在开头部分明确写出：

```lua
if a1 == 0 then
    return G[3147]
```

所以这里等价于：

```text
G[295] = G[3147]
```

这再次说明：

- 当前函数在“角色属性表/角色逻辑 ID”层面用的是 `0`
- 而不是 `99`

因此：

> `99` 更像样式编号；`0` 更像角色逻辑编号。

这是这次解析比前一篇新增出来的一个非常重要的认识。

## 5. 整体控制流拆解

整个函数可以拆成五个阶段。

---

## 5.1 阶段一：初始化主角/大雅名字栏样式

```lua
S0 = 99
G[227] = S0
S0 = G[227]
__ret = f_000019E0(S0)
S0 = 7
G[2000] = S0
```

### 含义

1. 把当前名字栏样式 ID 设为 `99`
2. 调用 `f_000019E0(99)` 初始化对应的名字栏字体/颜色/输出尺寸
3. 再把 `G[2000]` 强制设为 `7`

### 关于 `G[2000] = 7`

和前一个函数不同：

- 前一个函数是对 `G[2001]` 做条件推进
- 当前函数直接写死：`G[2000] = 7`

这说明：

> 对主角/大雅这一类名字显示，脚本不走渐进式状态推进，而是直接把对应状态置到一个固定模式值 `7`。

目前仅凭这一个函数，还不能完全确定 `7` 的最终语义，但至少可以判断：

- `G[2000]` 是主角/大雅这一套名字显示状态变量
- `7` 是一个“固定成熟状态”或“立即生效状态”

---

## 5.2 阶段二：根据 `a2` 决定显示哪一种名字

这里是整个函数最核心的逻辑。

### 分支一：`a2 == -1`

```lua
S0 = a2
S1 = -1
S0 = (S0 == S1)
```

若成立，则执行：

```lua
f_0004CCDA("　 ？？？ 　", "　 ？大雅 　")
f_000977B8(0, true)
```

### 含义

- 名字栏显示：`？？？`
- 备用/真实名：`？大雅`
- 用于“身份未公开”的场景

---

### 分支二：`a2 == 20`

```lua
S0 = a2
S1 = 20
S0 = (S0 == S1)
```

若成立，则执行：

```lua
f_0004CCDA("　　ぼく　　", nil)
f_000977B8(0, true)
```

### 含义

- 名字栏显示：`ぼく`
- 这是非常有代表性的特殊分支

它说明当前函数不仅处理“角色真实名是否公开”，还处理：

> **主角第一人称/自称显示模式**

在实际游戏语境里，这通常出现在：

- 主角视角 narration / 自白
- 未正式点名但以第一人称说话
- 特殊回忆或心理活动

所以 `a2 == 20` 可以暂时理解为：

```text
name_mode = SELF_PRONOUN_BOKU
```

---

### 分支三：其他情况

若前两者都不成立，则执行：

```lua
f_0004CCDA("　　大雅　　", nil)
f_000977B8(0, true)
```

### 含义

- 名字栏正常显示：`大雅`
- 这是主角/大雅名字已正常显示的标准分支

---

## 5.3 阶段三：设置名字栏有效标志

```lua
S0 = 1
G[29] = S0
```

这一点与前一个函数一致，说明：

```text
G[29] = 当前说话人名字栏处于有效显示状态
```

---

## 5.4 阶段四：根据 `a5` 决定是否保留 `a1` 到 `G[293]`

这是当前函数比前一个函数更有意思的一点。

对应代码：

```lua
S0 = a5
S1 = 1
S0 = (S0 ~= S1)
if __is_nil(S0) then goto BB_000019C9 end

::BB_000019C0::
S0 = nil
G[293] = S0
goto BB_000019CE

::BB_000019C9::
S0 = a1
G[293] = S0
```

### 高层等价

```lua
if a5 != 1 then
    G[293] = nil
else
    G[293] = a1
end
```

### 含义

与前一个函数不同，这里不是无条件地把某个参数保存到 `G[293]`，而是：

- 如果 `a5 != 1`，清空 `G[293]`
- 只有当 `a5 == 1` 时，才把 `a1` 保存进去

这说明：

> `a5` 是一个“是否保留/转发某个名字上下文参数”的开关。

它很可能控制：

- 当前这一句是否继承某种说话人附加状态
- 是否保留某个立绘/表情/语音上下文
- 是否把 `a1` 作为有效的名字栏辅助参数

---

## 5.5 阶段五：保存上下文并缓存角色属性

```lua
S0 = a3
G[294] = S0
S0 = 0
__ret = f_0008D409(S0)
S0 = __ret
G[295] = S0
```

### 高层含义

```text
G[294] = a3
G[295] = GetCharacterPropertyById(0)
```

即：

- 把调用上下文参数 `a3` 存入 `G[294]`
- 把角色逻辑编号 0 对应的属性值缓存到 `G[295]`

这里的 `0` 进一步强化了“主角/大雅的逻辑角色编号是 0”这一推测。

## 6. 还原成更容易读的伪代码

基于以上分析，这个函数可以近似还原成：

```lua
function ShowSpeakerName_TaigaLike(arg1, name_mode, arg3, keep_arg1_flag, arg5)
    -- 1. 使用主角/大雅的特殊名字栏样式
    G[227] = 99
    SetupSpeakerNameStyleByDisplayStyleId(99)

    -- 2. 把主角/大雅相关名字状态直接置为固定模式
    G[2000] = 7

    -- 3. 根据 name_mode 选择显示文本
    if name_mode == -1 then
        -- 身份未公开
        SetAndPrintSpeakerName("　 ？？？ 　", "　 ？大雅 　")
        SyncSpeakerState(0, true)
    elseif name_mode == 20 then
        -- 第一人称/自称显示
        SetAndPrintSpeakerName("　　ぼく　　", nil)
        SyncSpeakerState(0, true)
    else
        -- 正常真实名
        SetAndPrintSpeakerName("　　大雅　　", nil)
        SyncSpeakerState(0, true)
    end

    -- 4. 名字栏显示有效
    G[29] = 1

    -- 5. 根据开关决定是否保留 arg1
    if arg5 != 1 then
        G[293] = nil
    else
        G[293] = arg1
    end

    -- 6. 保存其他上下文与角色属性
    G[294] = arg3
    G[295] = GetCharacterPropertyById(0)
end
```

这里最重要的新信息有两个：

1. **这个函数是三态名字显示函数**
   - `？？？`
   - `ぼく`
   - `大雅`

2. **它揭示了两套编号系统**
   - `99`：名字栏样式/特殊显示类别编号
   - `0`：角色逻辑编号/属性编号

## 9. 结论

这段 `f_00001918` 的具体作用可以概括为：

> **为主角/大雅这一类特殊说话人设置名字栏样式，并根据参数在“？？？ / ぼく / 大雅”三种名字显示形态之间切换；同时刷新名字栏文本槽，设置名字栏有效状态，并有条件地缓存上下文参数与角色属性。**

如果用更贴近游戏表现的话说，它就是：

```text
主角（大雅）名字显示控制函数
```

并且它相比前一个角色名函数，额外揭示了一个重要机制：

> FVP 的“说话人名显示系统”并不只是“角色编号 -> 名字字符串”，而是至少分成了：
>
> 1. 名字栏样式编号（如 99）
> 2. 角色逻辑编号（如 0）
> 3. 名字显示模式参数（如 -1 / 20 / normal）
> 4. 名字栏上下文附加参数（如是否保留 `G[293]`）

这意味着，后续如果你要把名字栏系统完整抽象出来，至少要把这四层都建模进去，而不是只保留一个“speaker_name”字段。
