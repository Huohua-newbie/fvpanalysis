# Sakura.lua 第一处 function 说话人名显示逻辑详解

## 0. 解析对象

本文解析 `Sakura.lua` 中第一处函数：

```lua
function f_00000004(a0, a1, a2)
```

用户已指出：这一段在实际游戏中用于**控制文本框左侧说话人名的显示**。结合原始 HCB 字节串与 Lua-like IR，可以确认它确实是一个“角色名显示入口函数”，并且这里对应的角色是：

```text
クロ
```

也就是说，`f_00000004` 可以暂时命名为：

```text
ShowSpeakerName_クロ(...)
```

或更保守地命名为：

```text
SpeakerNameSlot_1_クロ(...)
```

第二个 `ret` 基本可以视为对齐/冗余/不可达安全返回块。

## 3. 字符串解码

这段中有三个关键字符串，均为 Shift-JIS 编码。

### 3.1 未知名显示字符串

原字节：

```text
81 40 20 81 48 81 48 81 48 20 81 40 00
```

解码为：

```text
　 ？？？ 　
```

含义：左侧说话人名显示成“？？？”。

---

### 3.2 未知状态下的真实/备用名

原字节：

```text
81 40 81 48 83 4E 83 8D 81 40 81 40 00
```

解码为：

```text
　？クロ　　
```

含义：带问号的“クロ”。

这很可能用于：

- 内部历史记录
- 语音/人物状态
- 真实名缓存
- 或者“尚未完全公开名字”的显示逻辑

---

### 3.3 正常名显示字符串

原字节：

```text
81 40 81 40 83 4E 83 8D 81 40 81 40 00
```

解码为：

```text
　　クロ　　
```

含义：左侧说话人名正常显示“クロ”。

注意前后有全角空格，这是为了让姓名在文本框左侧固定宽度区域中居中/对齐。

## 4. 关键子函数关系

该函数调用了几个重要子函数。

---

## 4.1 `f_000019E0(id)`：按角色编号初始化名字显示样式

调用位置：

```lua
S0 = G[227]
__ret = f_000019E0(S0)
```

在 `Sakura.lua` 中 `f_000019E0` 大致长这样：

```lua
function f_000019E0(a0)
  -- 根据 a1 / 角色编号分支
  -- 调用 TextOutSize
  -- 调用 TextColor
end
```

例如角色编号为 1 的分支中有：

```lua
S0 = 1
S1 = 52
__ret = f_0008CC08(S0, S1)
S0 = 8
S1 = 10
S2 = 52
S3 = 100
__ret = __syscall("TextColor", S0, S1, S2, S3)
S0 = 0
S1 = 5
S2 = nil
__ret = __syscall("TextOutSize", S0, S1, S2)
S0 = 0
S1 = 10
S2 = 52
S3 = 100
__ret = __syscall("TextColor", S0, S1, S2, S3)
```

所以它的功能可以概括为：

> 根据角色编号设置文本/名字显示用的颜色、字号或输出尺寸。

对当前函数来说：

```lua
G[227] = 1
f_000019E0(G[227])
```

说明 `f_00000004` 先把当前角色设为 1，再配置角色 1 的名字显示样式。

---

## 4.2 `f_0004CCDA(name, sub_name)`：设置并打印左侧说话人名

当前函数中两次调用：

### 未知名分支

```lua
S0 = "　 ？？？ 　"
S1 = "　？クロ　　"
__ret = f_0004CCDA(S0, S1)
```

### 正常名分支

```lua
S0 = "　　クロ　　"
S1 = nil
__ret = f_0004CCDA(S0, S1)
```

`f_0004CCDA` 的核心逻辑如下：

```lua
S0 = a1
G[233] = S0
S0 = a2
S1 = nil
S0 = (S0 == S1)
if __is_nil(S0) then goto BB_0004CD05 end

-- 第二参数为 nil 时：G[234] = G[233]
S0 = G[233]
G[234] = S0

goto BB_0004CD0A

-- 第二参数非 nil 时：G[234] = a2
S0 = a2
G[234] = S0

-- 如果 G[233] 非 nil，则 G[235] = G[233]
S0 = G[233]
S1 = nil
S0 = (S0 ~= S1)
if __is_nil(S0) then goto BB_0004CD1A end
S0 = G[233]
G[235] = S0

-- 清空并打印文本槽 8
S0 = 8
__ret = __syscall("TextClear", S0)
...
S0 = 8
S1 = G[233]
__ret = __syscall("TextPrint", S0, S1)
```

可见它的功能非常明确：

| 全局变量 | 作用推测 |
|---|---|
| `G[233]` | 当前实际要打印到名字框的文本 |
| `G[234]` | 备用名/真实名/历史名，如果第二参数为空则等于当前名 |
| `G[235]` | 最近一次非空说话人名缓存 |
| `Text slot 8` | 左侧说话人名显示区域 |

因此 `f_0004CCDA` 可以暂命名为：

```text
SetAndPrintSpeakerName(name, alt_name)
```

或：

```text
UpdateNameBoxText(name, alt_name)
```

## 4.3 `f_000977B8(id, true)`：角色名显示后的同步处理

当前函数中两次调用：

```lua
S0 = G[227]
S1 = true
__ret = f_000977B8(S0, S1)
```

`f_000977B8` 本身很短：

```lua
function f_000977B8(a0, a1)
  S0 = a1
  S1 = a2
  __ret = f_0009622F(S0, S1)
  return
end
```

也就是说它只是转调：

```text
f_0009622F(id, flag)
```

结合调用点，它大概率用于：

- 同步当前角色 ID
- 更新某种角色/语音/演出状态
- 通知名字框相关系统“当前名字已经刷新”

当前证据还不足以完全确定 `f_0009622F` 的具体含义，因此这里先保守命名为：

```text
SyncSpeakerState(id, flag)
```

---

## 4.4 `f_0008D409(id)`：按角色编号返回角色相关参数

当前函数末尾调用：

```lua
S0 = 1
__ret = f_0008D409(S0)
S0 = __ret
G[295] = S0
```

`f_0008D409` 是一个很长的分发表，开头如下：

```lua
if a1 == 0 then return G[3147]
if a1 == 1 then return G[3153]
if a1 == 2 then return G[3159]
if a1 == 3 then return G[3165]
...
```

当前传入的是 `1`，所以它会返回：

```text
G[3153]
```

并存入：

```text
G[295]
```

因此这里可以理解为：

> 根据当前说话人编号 1，取出该角色对应的某个状态/属性值，并缓存到 `G[295]`。

这个属性可能与：

- 角色语音
- 角色颜色
- 名字框样式
- 角色当前表情/立绘状态

有关。仅凭当前片段不能完全确定，但它确实是“角色编号 -> 角色属性表”的取值函数。

## 5. 整体控制流解析

整个函数可以拆成六个逻辑阶段。

---

## 5.1 阶段一：设置当前角色 ID

对应代码：

```lua
S0 = 1
G[227] = S0
S0 = G[227]
__ret = f_000019E0(S0)
```

含义：

```text
当前说话人 = 角色 1（クロ）
初始化角色 1 的名字显示样式
```

其中：

```text
G[227] = 当前说话人编号
```

在这个函数中固定为 `1`。

---

## 5.2 阶段二：根据 `G[2001]` 推进“クロ”的名字显示状态

对应代码：

```lua
if G[2001] in {0,2,4,6}:
    G[2001] = G[2001] + 1
```

Lua-like IR 中表现为一长串：

```lua
G[2001] == 0 or G[2001] == 2 or G[2001] == 4 or G[2001] == 6
```

如果为真，就执行：

```lua
G[2001] = G[2001] + 1
```

### 解释

`G[2001]` 很可能是“角色 1 / クロ”的名字显示状态计数器。

从后面的逻辑看，它不是简单布尔，而是一个分阶段编号。

---

## 5.3 阶段三：根据参数决定显示“？？？”还是“クロ”

关键判断：

```lua
S0 = a2
S1 = -1
S0 = (S0 == S1)
if __is_nil(S0) then goto BB_00000079 end
```

也就是：

```text
if 参数 == -1:
    使用未知名分支
else:
    使用正常名分支
```

### 未知名分支

```lua
f_0004CCDA("　 ？？？ 　", "　？クロ　　")
f_000977B8(G[227], true)
```

实际效果：

- 名字框左侧显示：`？？？`
- 同时保存/传入一个带问号的真实名：`？クロ`
- 用于“角色身份尚未公开”的场景

### 正常名分支

```lua
f_0004CCDA("　　クロ　　", nil)
f_000977B8(G[227], true)
```

实际效果：

- 名字框左侧显示：`クロ`
- 第二参数为空，表示没有额外备用名

---

## 5.4 阶段四：正常名分支下再次推进状态

正常名分支中还有一段：

```lua
if G[2001] in {0,1,4,5}:
    G[2001] = G[2001] + 2
```

也就是：

- 如果当前名字状态处在某些阶段；
- 并且本次是正常名显示；
- 则状态额外推进 2。

### 状态变化举例

假设进入函数前：

| 初始 `G[2001]` | 阶段二后 | 正常名分支阶段四后 | 可能含义 |
|---:|---:|---:|---|
| 0 | 1 | 3 | 从未显示 -> 已正常显示 |
| 1 | 1 | 3 | 某种半公开状态 -> 已正常显示 |
| 2 | 3 | 3 | 已经过渡过 -> 保持已显示 |
| 4 | 5 | 7 | 第二组状态推进到最终 |
| 5 | 5 | 7 | 第二组半公开状态 -> 最终 |
| 6 | 7 | 7 | 已经过渡过 -> 保持最终 |

如果走未知名分支（参数 `-1`），则只会进行阶段二的 `+1`，不会进行阶段四的 `+2`。

这非常符合“人物名公开/未公开状态机”的结构：

- 未知显示与正常显示会使状态进入不同阶段；
- 角色名公开后会把状态推进到另一个区间。

---

## 5.5 阶段五：写入当前文本/说话人上下文全局变量

末尾部分：

```lua
S0 = 1
G[29] = S0
S0 = a1
G[293] = S0
S0 = a3
G[294] = S0
```

含义推测：

| 全局变量 | 作用推测 |
|---|---|
| `G[29] = 1` | 当前有有效说话人名 / 当前文本框名显示启用 |
| `G[293] = a1` | 保存调用者传入的某个上下文参数 |
| `G[294] = a3` | 保存调用者传入的另一个上下文参数 |

这里的 `a1/a3` 是 Lua-like IR 对栈帧参数的命名结果。由于底层 HCB 是通过 `push_stack -4`、`push_stack -2` 取参数槽，名字不一定完全符合普通 Lua 的参数编号直觉。应以“保存调用参数到全局上下文”理解。

---

## 5.6 阶段六：缓存角色属性到 `G[295]`

最后：

```lua
S0 = 1
__ret = f_0008D409(S0)
S0 = __ret
G[295] = S0
```

即：

```text
G[295] = GetCharacterProperty(1)
```

而 `f_0008D409(1)` 对应返回 `G[3153]`。

所以：

```text
G[295] = G[3153]
```

至少在角色 1 的情况下是这样。

这说明 `G[295]` 是一个“当前说话人关联属性缓存位”。

## 6. 还原成更容易读的伪代码

基于以上分析，可以把整个函数改写成下面这种伪代码：

```lua
function ShowSpeakerName_Kuro(arg_context_1, reveal_mode, arg_context_2)
    -- 1. 设置当前说话人编号
    G[227] = 1                  -- 角色 1 = クロ
    SetupSpeakerNameStyle(G[227])

    -- 2. 第一次推进クロ的名字显示状态
    if G[2001] == 0 or G[2001] == 2 or G[2001] == 4 or G[2001] == 6 then
        G[2001] = G[2001] + 1
    end

    -- 3. 根据 reveal_mode 判断显示“？？？”还是“クロ”
    if reveal_mode == -1 then
        -- 身份未知/隐藏状态
        SetAndPrintSpeakerName("　 ？？？ 　", "　？クロ　　")
        SyncSpeakerState(G[227], true)
    else
        -- 正常显示角色名
        SetAndPrintSpeakerName("　　クロ　　", nil)
        SyncSpeakerState(G[227], true)

        -- 4. 正常名显示后进一步推进状态
        if G[2001] == 0 or G[2001] == 1 or G[2001] == 4 or G[2001] == 5 then
            G[2001] = G[2001] + 2
        end
    end

    -- 5. 写入当前文本/说话人上下文
    G[29] = 1
    G[293] = arg_context_1
    G[294] = arg_context_2

    -- 6. 缓存当前角色属性
    G[295] = GetCharacterProperty(1)
end
```

## 9. 关于 `a1/a2/a3` 的注意事项

Lua-like IR 中函数写作：

```lua
function f_00000004(a0, a1, a2)
```

但正文中出现：

```lua
S0 = a3
```

这是当前 HCB -> Lua-like IR 栈帧参数命名方式带来的现象。底层 HCB 实际使用的是：

```text
push_stack -3
push_stack -4
push_stack -2
```

这些负数偏移代表“从当前栈帧向上取调用参数”。由于 converter 的命名规则和普通 Lua 参数列表并不完全一致，所以这里不宜把 `a0/a1/a2/a3` 当作普通 Lua 函数参数严格理解。

更稳妥的说法是：

- `push_stack -3`：用于判断是否显示未知名，Lua 中显示为 `a2`
- `push_stack -4`：末尾保存到 `G[293]`，Lua 中显示为 `a1`
- `push_stack -2`：末尾保存到 `G[294]`，Lua 中显示为 `a3`

也就是说，它们都是调用者传来的上下文参数，只是当前 Lua-like 输出的名字还不是最终高级语义名。

## 10. 结论

这段首个 function 的具体作用可以概括为：

> **设置当前说话人为クロ，并根据传入参数决定文本框左侧显示“？？？”还是“クロ”；同时更新クロ的名字显示/公开状态计数器，刷新名字栏文本槽，设置当前说话人相关全局状态，并缓存该角色的属性值。**

如果按实际游戏功能命名，它大致就是：

```text
ShowSpeakerName_Kuro(reveal_mode, context...)
```

其中最核心的显示逻辑是：

```text
参数 == -1:
    左侧名字栏显示 “？？？”
    备用/真实名记录为 “？クロ”
否则:
    左侧名字栏显示 “クロ”
```

而 `G[2001]` 的状态推进说明：

> 游戏不仅仅是“打印一个名字”，还在记录该角色名是否已经以未知/正常方式显示过，这很可能与身份公开、回想、历史文本或后续演出状态有关。

这也说明：

- `Sakura.lua` 中这种角色名显示函数并非孤立文本打印；
- 它是一个包含“角色编号、名字栏样式、未知名/正常名分支、状态计数、上下文缓存”的完整角色说话人名控制模块。
