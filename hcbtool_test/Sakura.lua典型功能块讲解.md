# Sakura.lua 典型功能块解析讲解稿

## 0. 说明

本文基于：

- `fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua`
- `fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.cfg.json`
- `fvp_analysis/result/hcb可逆转换/指令与指令块功能对照表.md`
- `fvp_analysis/result/syscall语义数据库/syscall_spec.json`

暂时只着眼于 `Sakura.lua` 这个 Lua-like IR 文件，对其中**典型功能块**做讲解。

这里的“功能块”不是严格的编译器术语，而是从**脚本功能理解**角度出发，把若干条指令组合在一起看成一个“实现某件事的最小单元”。

讲解原则：

- 每种功能块只挑一个代表性例子；
- 先给原始片段；
- 再拆解每一步在做什么；
- 最后说明它在游戏中的实际意义。

## 1. 条件分支 + 状态计数块

### 例子
来自 `Sakura.lua` 开头函数 `f_00000004`：

```lua
S0 = G[2001]
S1 = 0
S0 = (S0 == S1)
S1 = G[2001]
S2 = 2
S1 = (S1 == S2)
S0 = ((S0 ~= nil) or (S1 ~= nil))
S1 = G[2001]
S2 = 4
S1 = (S1 == S2)
S0 = ((S0 ~= nil) or (S1 ~= nil))
S1 = G[2001]
S2 = 6
S1 = (S1 == S2)
S0 = ((S0 ~= nil) or (S1 ~= nil))
if __is_nil(S0) then goto BB_0000003D end

::BB_00000034::
S0 = G[2001]
S1 = 1
S0 = (S0 + S1)
G[2001] = S0
```

### 分步解释

#### 第一步：读取一个全局状态值
```lua
S0 = G[2001]
```
这里把全局变量 `G[2001]` 取到操作数栈变量 `S0` 中。

在 FVP 里，很多剧情状态、计数器、模式位都存在全局变量区里，所以 `G[2001]` 很可能是某个“角色显示状态”“剧情阶段编号”或“轮换计数”。

#### 第二步：依次比较它是否等于 0、2、4、6
```lua
S0 = (S0 == S1)
...
S1 = (S1 == S2)
S0 = ((S0 ~= nil) or (S1 ~= nil))
```
这一串结构其实是在做：

```text
G[2001] == 0 or G[2001] == 2 or G[2001] == 4 or G[2001] == 6
```

由于 HCB 是栈机，所以它不是直接写一个高级条件表达式，而是：

- 先做一次比较；
- 再做下一次比较；
- 再把多个比较结果用 `or` 串起来。

#### 第三步：条件为真则进入计数递增块
```lua
if __is_nil(S0) then goto BB_0000003D end
```
这里要注意 FVP 的真假语义：

- `Nil` 为假
- 非 `Nil` 为真

所以这句等价于：

```text
如果上面那一长串 or 条件为假，则跳过加一逻辑；
如果为真，则继续执行 BB_00000034。
```

#### 第四步：把全局状态值加 1
```lua
S0 = G[2001]
S1 = 1
S0 = (S0 + S1)
G[2001] = S0
```
这一块就是很标准的：

```text
G[2001] = G[2001] + 1
```

### 这个功能块的整体含义
这是一个非常典型的：

> **“根据若干个状态值范围/离散点，决定是否推进一个计数器”**

块。

这种模式在 AVG/视觉小说脚本里很常见，常用于：

- 人物表情轮换索引
- 立绘差分切换编号
- UI 菜单页签轮换
- 某个剧情子状态推进

它本质上是一个**离散状态机更新块**。

## 2. 文本打印块

### 例子
来自 `Sakura.lua` 第 `154629~154631` 行附近：

```lua
S0 = 8
S1 = G[233]
__ret = __syscall("TextPrint", S0, S1)
```

### 分步解释

#### 第一步：指定文本缓冲槽
```lua
S0 = 8
```
这里的 `8` 不是文字内容，而更像是：

- 某个文本缓冲槽 ID
- 或某个文字层/窗口编号

在 `rfvp` 的 `TextPrint` 相关实现里，第一参数通常就是 **text buffer id**。

#### 第二步：取出要显示的文本
```lua
S1 = G[233]
```
这里从全局变量 `G[233]` 里取出实际文本内容。

这说明在 `Sakura.lua` 中，文本来源并不总是直接内联 `push_string`，也可能已经被前面的逻辑预存到全局变量里，再由打印块统一调度输出。

#### 第三步：调用 TextPrint
```lua
__ret = __syscall("TextPrint", S0, S1)
```
`TextPrint` 的语义不是简单“print 一下字符串”，而通常包括：

- 把文本送进 TextManager
- 更新当前文本缓冲
- 可能触发逐字显示逻辑
- 可能触发 `text_wait`
- 当前脚本线程可能因此让出执行

### 这个功能块的整体含义
这是最典型、最核心的：

> **“把某段文本送到指定文本窗口/文本缓冲去显示”**

块。

在游戏层面上，它几乎就是“真正把一句台词/旁白打到屏幕上”的动作。

## 3. 文本打印 + 线程让出 + 文本清空循环块

### 例子
来自 `Sakura.lua` 第 `291731~291740` 行附近：

```lua
S0 = 1
if __is_nil(S0) then goto BB_0009459F end
::BB_0009454A::
S0 = G[1827]
S1 = "そのほんの少しの隙間から、どこからか小さな春色が舞い込んで――"
__ret = __syscall("TextPrint", S0, S1)
__ret = __syscall("ThreadNext")
S0 = G[1827]
__ret = __syscall("TextClear", S0)
goto BB_00094543
```

### 分步解释

#### 第一步：强制进入循环体
```lua
S0 = 1
if __is_nil(S0) then goto ...
```
由于 `S0 = 1`（非 `Nil`），因此这段判断恒真，相当于一个人为构造的：

```text
while true do
```

循环骨架。

#### 第二步：打印一行固定文本
```lua
S0 = G[1827]
S1 = "そのほんの少しの隙間から、どこからか小さな春色が舞い込んで――"
__ret = __syscall("TextPrint", S0, S1)
```
这里把固定字符串送到 `G[1827]` 指定的文本槽位去打印。

#### 第三步：让出线程
```lua
__ret = __syscall("ThreadNext")
```
`ThreadNext` 的作用是：

- 当前 context 主动让出执行权
- 调度器切到下一个可运行线程/上下文

也就是说，这里不是一口气在同一帧内执行完，而是：

> 打印一句 -> 把时间交给别的逻辑/下一帧 -> 再回来继续。

#### 第四步：清空文本
```lua
S0 = G[1827]
__ret = __syscall("TextClear", S0)
```
这里清掉刚才打印的文本槽内容。

#### 第五步：跳回循环头
```lua
goto BB_00094543
```
于是形成：

```text
打印 -> 让出执行 -> 清空 -> 再打印 -> ...
```

### 这个功能块的整体含义
这是一个非常典型的：

> **“循环闪烁/循环刷新文本效果块”**

它不像普通 TextPrint 那样“打一行剧情然后结束”，而更像：

- 滚动提示
- 演出特效文本
- 瞬时提示反复刷新
- 某种过场/诗句/屏幕特效字幕

这个例子很好地说明：

- 不能把 `TextPrint` 单独拿出来理解；
- 还要看它后面是否跟了 `ThreadNext`、`ThreadWait`、`TextClear`；
- 多条组合之后，才形成真正的“演出逻辑”。

## 4. 图像加载块

### 例子
来自 `Sakura.lua` 第 `118943~118952` 行附近：

```lua
S0 = "graph/"
a0 = S0
S0 = a0
S1 = a3
S0 = (S0 + S1)
a0 = S0
S0 = a2
S1 = a0
__ret = __syscall("GraphLoad", S0, S1)
```

### 分步解释

#### 第一步：构造资源路径前缀
```lua
S0 = "graph/"
a0 = S0
```
先把固定前缀 `graph/` 放入参数槽里。

#### 第二步：与后缀名拼接
```lua
S0 = a0
S1 = a3
S0 = (S0 + S1)
a0 = S0
```
这里把：

```text
a0 = "graph/" + a3
```

也就是构造一个完整资源路径，例如：

- `graph/BGS002_parts1`
- `graph/title_logo`
- `graph/snow000`

#### 第三步：执行 GraphLoad
```lua
S0 = a2
S1 = a0
__ret = __syscall("GraphLoad", S0, S1)
```
第一参数是 graph 槽位 ID，第二参数是图像路径。

### 这个功能块的整体含义
这是一个标准的：

> **“拼出图像资源路径并加载到某个 graph 槽位”**

块。

它通常是后续所有图像显示逻辑的前置步骤。后面如果要把图像真正画出来，往往还需要接：

- `PrimSetSprt`
- `PrimSetXY`
- `PrimSetWH`
- `PrimSetZ`
- `PrimSetAlpha`
- `PrimSetBlend`

等图元设置指令。

## 5. 图元绑定块（把 graph 真正挂到 prim 上）

### 例子
来自 `Sakura.lua` 第 `118955~118957` 行附近：

```lua
S0 = a1
S1 = a2
S2 = nil
S3 = nil
__ret = __syscall("PrimSetSprt", S0, S1, S2, S3)
```

### 分步解释

#### 第一步：准备 prim id 与 graph id
```lua
S0 = a1
S1 = a2
```
- `S0`：目标 prim ID
- `S1`：前面 `GraphLoad` 载入的 graph ID

#### 第二步：附加两个可选参数，当前为空
```lua
S2 = nil
S3 = nil
```
这说明这个 syscall 在这里使用的是一种“默认参数”方式。

#### 第三步：调用 PrimSetSprt
```lua
__ret = __syscall("PrimSetSprt", S0, S1, S2, S3)
```
含义是：

> 把 `graph id` 对应的贴图，绑定到 `prim id` 对应的显示对象上。

### 这个功能块的整体含义
这是典型的：

> **“贴图绑定块”**

单独 `GraphLoad` 只是把资源放进缓存/graph 槽位；
真正让它“上屏”的关键，往往是 `PrimSetSprt` 这一类调用。

也就是说：

- `GraphLoad` 更偏资源层
- `PrimSetSprt` 更偏显示层

这两个连在一起，才构成“显示一张图”的骨架。

## 6. 图元属性设置块（透明度 / 混合 / 中心点 / Z / 缩放）

### 例子
来自 `Sakura.lua` 第 `39145~39193` 行附近：

```lua
S0 = 240
S1 = "BGS002_parts1"
S2 = 11
S3 = 2
__ret = f_000373A5(S0, S1, S2, S3)
S0 = 241
S1 = "BGS002_parts2"
S2 = 11
S3 = 2
__ret = f_000373A5(S0, S1, S2, S3)
S0 = 240
S1 = 128
__ret = __syscall("PrimSetAlpha", S0, S1)
S0 = 240
S1 = 1
__ret = __syscall("PrimSetBlend", S0, S1)
S0 = 241
S1 = 1
__ret = __syscall("PrimSetBlend", S0, S1)
S0 = 240
S1 = 1280
S2 = 2
S1 = (S1 / S2)
S2 = 720
S3 = 2
S2 = (S2 / S3)
__ret = __syscall("PrimSetOP", S0, S1, S2)
...
S0 = 240
S1 = 800
__ret = __syscall("PrimSetZ", S0, S1)
S0 = 241
S1 = 1000
__ret = __syscall("PrimSetZ", S0, S1)
...
S0 = 240
S1 = nil
S2 = 2000
__ret = __syscall("PrimSetRS", S0, S1, S2)
```

### 分步解释

#### 第一步：通过内部辅助函数加载并挂载两个部件图像
```lua
__ret = f_000373A5(S0, S1, S2, S3)
```
虽然这里不是直接 syscall，而是一个内部函数，但从名字和参数结构可以看出，它大概率封装了：

- 构造 `graph/...` 路径
- `GraphLoad`
- `PrimSetSprt`
- 某些附加设置

也就是说，它相当于一个“子功能块封装调用”。

#### 第二步：设置透明度
```lua
__ret = __syscall("PrimSetAlpha", 240, 128)
```
- `240` 是 prim id
- `128` 是 alpha 值

#### 第三步：设置混合模式
```lua
__ret = __syscall("PrimSetBlend", 240, 1)
```
表示 prim 使用某种 blend 模式。

#### 第四步：设置 OP（通常可理解为显示基准点/原点）
```lua
__ret = __syscall("PrimSetOP", S0, S1, S2)
```
这里把：

- `1280 / 2`
- `720 / 2`

算出来作为参数，说明是把 prim 的某个原点设置到屏幕中心附近。

#### 第五步：设置 Z 顺序
```lua
__ret = __syscall("PrimSetZ", 240, 800)
__ret = __syscall("PrimSetZ", 241, 1000)
```
让两个 prim 在画面中有前后层级。

#### 第六步：设置 RS（通常可理解为大小/缩放相关）
```lua
__ret = __syscall("PrimSetRS", S0, nil, 2000)
```
表示再对 prim 做某种缩放/尺寸设定。

### 这个功能块的整体含义
这是一整套非常完整的：

> **“演出图元配置块”**

它的作用不是“只显示一张图”，而是：

- 先装载两个图元部件
- 再配置透明、混合方式、中心点、Z 顺序、缩放
- 为接下来的复杂演出效果做准备

这种块常用于：

- CG 分层部件
- 特效背景部件
- UI 组合件
- 需要做动画/渐变/叠加的图元组

## 7. 音频加载与播放块

### 例子
来自 `Sakura.lua` 第 `124444~124454` 行附近：

```lua
S2 = G[1778]
S3 = a0
S2 = (S2 + S3)
S3 = 8
__ret = __syscall("IntToText", S2, S3)
S2 = __ret
S1 = (S1 + S2)
__ret = __syscall("AudioLoad", S0, S1)
S0 = G[218]
S1 = nil
__ret = __syscall("AudioPlay", S0, S1)
```

### 分步解释

#### 第一步：生成数字编号文本
```lua
S2 = G[1778]
S3 = a0
S2 = (S2 + S3)
S3 = 8
__ret = __syscall("IntToText", S2, S3)
S2 = __ret
```
这表示：

- 先取一个基础编号 `G[1778]`
- 再加上参数 `a0`
- 再通过 `IntToText(..., 8)` 转成 8 位字符串

这个模式很像在拼：

- `00000073`
- `00000074`
- 类似的音频编号字符串

#### 第二步：拼接成音频路径
```lua
S1 = (S1 + S2)
```
这里说明前面 `S1` 里大概率已经有某个前缀，如：

- `BGM/`
- `VOICE/`
- `SE/`

与编号字符串拼起来形成完整路径。

#### 第三步：加载音频
```lua
__ret = __syscall("AudioLoad", S0, S1)
```
把路径对应音频加载到某个 audio channel。

#### 第四步：播放音频
```lua
S0 = G[218]
S1 = nil
__ret = __syscall("AudioPlay", S0, S1)
```
然后使用 `AudioPlay` 让该 channel 开始播放。

### 这个功能块的整体含义
这是典型的：

> **“按编号动态构造 BGM/语音资源名并播放”**

块。

比“直接写死一个文件名”的音频块更灵活，说明脚本可能是在：

- 根据剧情状态选不同 BGM
- 根据角色或序号选不同语音
- 根据参数动态构造资源路径

## 8. 线程启动块

### 例子
来自 `Sakura.lua` 第 `39135~39137` 行附近：

```lua
S0 = 10
S1 = 91872
__ret = __syscall("ThreadStart", S0, S1)
```

### 分步解释

#### 第一步：指定线程槽位
```lua
S0 = 10
```
表示目标线程 ID 是 `10`。

#### 第二步：提供函数地址
```lua
S1 = 91872
```
这里的 `91872` 不是普通数值，而是：

> **HCB 代码区中的函数入口地址**

这也是为什么在回编译 HCB 时，这类 `push_i32` 必须参与重定位修正。

#### 第三步：启动线程
```lua
__ret = __syscall("ThreadStart", S0, S1)
```
表示在线程槽 `10` 中启动一个新的脚本上下文，从地址 `91872` 开始执行。

### 这个功能块的整体含义
这是非常典型的：

> **“并行子流程启动块”**

在游戏层面，它可能用于：

- 独立特效流程
- 并行演出更新
- 定时器/轮询子线程
- 一边显示主剧情，一边跑辅助 UI / 特效逻辑

这是理解 FVP“多线程”能力的关键例子：

- 它并不是 OS thread；
- 而是新建一个脚本 context。

## 9. 时间等待块

### 例子
来自 `Sakura.lua` 第 `119349~119350` 行附近：

```lua
S0 = a1
__ret = __syscall("ThreadWait", S0)
```

### 分步解释

#### 第一步：取等待时间参数
```lua
S0 = a1
```
把当前函数参数 `a1` 当作等待时间。

#### 第二步：调用 ThreadWait
```lua
__ret = __syscall("ThreadWait", S0)
```
其效果是：

- 当前线程进入 `WAIT` 状态
- 一段时间后由调度器恢复
- 当前函数不会在这一帧继续向下跑

### 这个功能块的整体含义
这是最标准的：

> **“显式时间等待块”**

常用于：

- 过场节奏控制
- 延迟显示下一句台词
- 动画/特效之间插入时间间隔

## 10. 窗口模式控制块

### 例子
来自 `Sakura.lua` 第 `123726~123749` 行附近：

```lua
S0 = a2
G[3529] = S0
S0 = a2
__ret = __syscall("WindowMode", S0)
::BB_00039CE4::
S0 = 2
__ret = __syscall("WindowMode", S0)
S0 = __ret
S1 = 0
S0 = (S0 == S1)
if __is_nil(S0) then goto BB_00039CF7 end
::BB_00039CF2::
S0 = 1
__ret = __syscall("WindowMode", S0)
...
S0 = a2
G[3142] = S0
S0 = true
G[1703] = S0
```

### 分步解释

#### 第一步：记录请求的窗口模式
```lua
S0 = a2
G[3529] = S0
```
把参数 `a2` 保存到全局变量里，说明脚本不只是“立刻切窗口模式”，还把请求状态持久化了。

#### 第二步：调用 WindowMode 设置/查询
```lua
S0 = a2
__ret = __syscall("WindowMode", S0)
```
`WindowMode` 在 FVP 里既可能是：
- 查询当前状态
- 设置窗口模式
- 查询某个模式是否允许

取决于传入 mode 值。

#### 第三步：检查返回值
```lua
S0 = 2
__ret = __syscall("WindowMode", S0)
S0 = __ret
S1 = 0
S0 = (S0 == S1)
if __is_nil(S0) then goto ...
```
说明这里不是一次简单调用，而是：

> **调用 WindowMode -> 再读取返回值 -> 再按结果分支处理**

#### 第四步：按结果修正全局状态
```lua
S0 = a2
G[3142] = S0
S0 = true
G[1703] = S0
```
说明窗口模式变化会联动多个全局标志位。

### 这个功能块的整体含义
这是一个典型的：

> **“窗口模式状态机块”**

它不只是“切全屏/窗口”，还包括：

- 记录目标模式
- 查询/验证结果
- 按结果更新脚本层全局状态
- 触发后续公共收尾逻辑

所以如果以后你想“自由创作”并让脚本控制窗口系统，这种块就是重要模板。

## 11. 资源清理 / 卸载块

### 例子
来自 `Sakura.lua` 第 `118833~118839` 行附近：

```lua
S0 = a1
__ret = __syscall("PrimSetNull", S0)
S0 = a1
S1 = nil
__ret = __syscall("GraphLoad", S0, S1)
S0 = a1
S1 = a0
__ret = __syscall("GraphLoad", S0, S1)
```

### 分步解释

#### 第一步：先把 prim 置空
```lua
__ret = __syscall("PrimSetNull", S0)
```
表示让某个 prim 不再显示原来的贴图/内容。

#### 第二步：用 `GraphLoad(..., nil)` 卸载旧图
```lua
S1 = nil
__ret = __syscall("GraphLoad", S0, S1)
```
在 FVP 里，`GraphLoad(index, nil)` 常常表示：

- 卸载该 graph 槽位上的资源
- 释放或清空对应图像内容

#### 第三步：再加载新图
```lua
S1 = a0
__ret = __syscall("GraphLoad", S0, S1)
```
把新路径对应图像载入同一个 graph 槽位。

### 这个功能块的整体含义
这是很标准的：

> **“先解绑/清空旧资源，再重载新资源”**

块。

它常用于：

- 替换当前显示贴图
- 人物表情/立绘切换
- 某个图元 slot 的动态刷新

相比直接 `GraphLoad(new_path)`，这种写法更保守，说明脚本作者或原始编译产物希望显式地清理旧状态。

## 12. 子函数封装块

### 例子
来自 `Sakura.lua` 第 `118905~118916` 行附近：

```lua
function f_000373A5(a0, a1, a2, a3)
  ...
  S0 = a1
  S1 = a2
  S2 = a4
  __ret = f_00037294(S0, S1, S2)
  S0 = a1
  S1 = nil
  S2 = nil
  __ret = f_00037345(S0, S1, S2)
  ...
end
```

### 分步解释

这不是 syscall，而是一个内部 helper function。

可以看出：

- 它不是直接承担最终剧情意义；
- 而是把若干底层操作封装成一个“可复用功能块”；
- 里面又进一步调用别的内部函数 `f_00037294`、`f_00037345`。

结合它周边被调用的上下文（如上面的 `BGS002_parts1/BGS002_parts2`），很可能这是一个：

- 立绘/部件装载 helper
- 贴图+图元初始化 helper

### 这个功能块的整体含义
这是典型的：

> **“脚本级封装块”**

它说明 HCB 不是只有“直接 syscall 的线性脚本”，而是已经编译成了：

- 若干内部函数
- 有调用关系
- 可复用的子流程库

这对“完全解析 HCB 作用”非常重要，因为以后不能只盯 syscall，还要识别：

- 哪些内部函数本身就代表了某个高层语义模板。

## 13. 总结：Sakura.lua 里这些典型块说明了什么

从以上例子可以看出，`Sakura.lua` 中至少稳定出现了这些高频功能单元：

1. **状态计数/状态推进块**
   - 通过全局变量 + 比较 + 加法实现小型状态机

2. **文本打印块**
   - 通过 `TextPrint` 输出台词/旁白/提示文字

3. **文本刷新循环块**
   - `TextPrint + ThreadNext + TextClear + goto` 形成重复闪烁/轮询显示

4. **图像加载块**
   - 拼路径 -> `GraphLoad`

5. **图元绑定块**
   - `PrimSetSprt` 把图像真正绑定到显示对象

6. **图元属性配置块**
   - `PrimSetAlpha/Blend/OP/Z/RS` 构成完整的演出配置单元

7. **音频播放块**
   - `IntToText` + 路径构造 + `AudioLoad` + `AudioPlay`

8. **线程启动块**
   - `ThreadStart(thread_id, function_addr)` 启动并行脚本 context

9. **等待块**
   - `ThreadWait` 控制节奏

10. **窗口模式控制块**
    - `WindowMode` 与全局状态联动，形成窗口模式状态机

11. **资源清理/卸载块**
    - `PrimSetNull + GraphLoad(nil) + GraphLoad(new)`

12. **子函数封装块**
    - 某些内部函数本身就是高层功能模板的“脚本库函数”

也就是说，Sakura.lua 并不是“杂乱的一堆底层指令”，而是已经能看出非常明显的：

> **视觉小说引擎脚本模板化编程风格**。

如果下一步继续深入，最值得做的是：

- 把这些功能块提炼成“模式库”；
- 进一步为每一类块自动识别并加上标签；
- 最终把 `Sakura.lua` 这样的大文件变成“按功能块索引”的分析结果。

这将非常有利于你后续实现：

- HCB 的完整语义解释
- 可视化脚本理解
- 自由创作时的“高层功能模板复用”
