# f_00002403 逐条反汇编对照

## 1. 基本结论

从用户提供的原始字节与 [`Sakura.lua`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua) 中的 Lua-like 反编译结果来看，`f_00002403` 是一个**很薄的包装函数**：

1. 先把 `G[2124]` 置为 `true`
2. 调用 `f_00010271()` 做一轮场景/显示相关初始化
3. 再把字符串资源名 `"KURO_e011a1"` 连同若干参数传给 `f_00051420(...)`
4. 最后调用 `f_00010298(nil, nil, a6)` 做后处理
5. 返回

结合：

- 资源名 `KURO_e011a1` 是实际存在的 CG 图名
- 后续紧邻的 [`f_0000243B`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:3672)、[`f_00002473`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:3697) 只是把资源名改成 `KURO_e011b1`、`KURO_e011c1`
- 同时分别写入不同全局标志 `G[2125]`、`G[2126]`

因此，这一组函数**高度疑似属于 CG 引入 / CG 差分切换 / 立绘或事件图显示包装器**。

更准确地说：

- [`f_00051420()`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:163489) 很可能是“图像/CG 显示核心函数”
- [`f_00002403`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:3647) 则是“某一张具体 CG `KURO_e011a1` 的专用入口包装”

## 2. 参数命名的一个重要注意点

Lua-like 导出中函数头是：

```lua
function f_00002403(a0, a1, a2, a3, a4, a5)
```

但函数体里却访问了 `a6`：

```lua
S2 = a6
```

这不是原函数真的有 7 个显式参数，而是当前 IR 导出器对 `push_stack` 负偏移的参数命名存在一个**整体偏移 1** 的现象。

按原始字节看：

- `10 F9` = `push_stack -7`
- `10 FA` = `push_stack -6`
- `10 FB` = `push_stack -5`
- `10 FC` = `push_stack -4`
- `10 FD` = `push_stack -3`
- `10 FE` = `push_stack -2`

在导出 Lua 中，这些分别被显示成：

- `a1`
- `a2`
- `a3`
- `a4`
- `a5`
- `a6`

所以在阅读 [`f_00002403()`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:3647) 时，更合理的理解是：

> 这个函数实际消耗的是 **6 个传入值**，只是 Lua-like 文本中的参数名整体右移了 1 位。

也就是说，这里的 `a1..a6`，本质上就是“6 个实参槽”。

### 6.2 [`f_00051420()`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:163489)
这是最关键的核心函数：

- 参数多达 10 个
- 第一参数是资源名字符串
- 内部会设置 `G[118] = true`、`G[16] = 1`
- 会根据 `a3` 设置 `G[164]`
- 会给多个参数做 `nil -> 默认值` 归一化
- 后面还有大量显示流程控制

因此，它极有可能是：

> **通用的图像/CG/立绘显示入口**

而 [`f_00002403`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:3647) 只是把某一张具体资源名和固定参数模板喂给它。

### 6.3 [`f_00010298()`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:29117)
该函数有 3 个参数，会先调用 `f_00096D1B()`，随后检查：

并对传入参数做默认值修正。

它更像是**显示后的模式修正 / 过渡 / 等待 / 收尾控制函数**，而不像实际资源装载核心。

## 8. 当前判断

以第一处 [`f_00002403`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:3647) 为例，你的判断**非常有道理，而且目前证据相当强**：

- 资源名直接就是实际 CG 名
- 存在连续的大量同模板函数，只替换资源名和解锁/标记用的全局变量
- 中心调用 [`f_00051420()`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:163489) 是一个明显的通用显示大函数
- 调用前还有统一初始化函数 [`f_00010271()`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:29097)
- 调用后还有统一后处理函数 [`f_00010298()`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:29117)

所以，当前可以暂时把 [`f_00002403`](fvp_analysis/result/hcbtool_test/Sakura_hcb_ir/Sakura.lua:3647) 视为：

> **CG `KURO_e011a1` 的显示包装函数**

而不是独立的底层渲染核心。

更上层的族判断则是：

> `0x00002403` 开始的一大段相似函数，极可能是“按资源名分拆好的 CG 引入入口函数族”。
