# HCB 指令与指令块功能对照表

## 0. 文档目的

本文档配合 `hcb_ir_core.py` / `hcb_to_ir.py` / `ir_to_hcb.py` 使用，面向人类读者总结：

1. 每一种 HCB 指令大致做了什么；
2. 多条指令如何组成“实现特定功能的单个指令块”；
3. 这些块在转换成 CFG / Lua-like IR 后大概会长成什么样。

本文档以：

- `rfvp-0.3.0/crates/rfvp/src/script/context.rs`
- `rfvp-0.3.0/crates/hcb2lua_decompiler/*`
- `rfvp-0.3.0/crates/lua2hcb_compiler/*`

为主要参考。

---

## 1. 先理解：HCB 是栈机，不是寄存器机

HCB 中很多指令看似“没有参数”，其实它们都默认操作**操作数栈**顶端的数据。

例如：

- `push_*`：压栈
- `add/sub/...`：弹两个值做运算，再把结果压回去
- `jz`：弹栈顶作为条件，若假则跳转
- `syscall`：弹出若干参数调用宿主，再把结果写进 `__ret`

所以分析单条指令时，不要只看它自己，还要看它前面几条 `push_*` 准备了什么。

---

## 2. 单指令功能对照

## 2.1 控制流与调用类

### `init_stack args locals`
**作用**：
- 标记一个函数开始
- 声明当前函数参数个数
- 为局部变量分配栈空间

**典型意义**：
- 基本可以把它视作“函数序言”

**Lua-like 近似**：
```lua
-- init_stack args=2 locals=3
```

---

### `call target`
**作用**：
- 调用 HCB 内部函数
- 保存返回地址和当前帧信息
- 跳转到目标地址执行

**结果**：
- 返回值不会直接压栈，而是先存到 `__ret` / return register
- 若脚本随后需要它，通常会用 `push_return`

**Lua-like 近似**：
```lua
__ret = f_XXXXXXXX(arg0, arg1, ...)
```

---

### `syscall id`
**作用**：
- 根据导入表序号查出 syscall 名称
- 按该 syscall 的参数个数从栈里弹参数
- 调用宿主引擎功能
- 结果写入 `__ret`

**Lua-like 近似**：
```lua
__ret = __syscall("TextPrint", S0, S1)
```

---

### `ret`
**作用**：
- 从当前函数返回
- 返回值设为 `Nil`

### `retv`
**作用**：
- 把栈顶值作为返回值返回

**Lua-like 近似**：
```lua
return
```
或
```lua
return S0
```

---

### `jmp target`
**作用**：
- 无条件跳转

### `jz target`
**作用**：
- 弹出栈顶条件值
- 若为假（`Nil`）则跳转

**Lua-like 近似**：
```lua
if __is_nil(S0) then goto BB_xxxxxxxx end
```

---

## 2.2 常量与取值类

### `push_nil`
压入假值 `Nil`

### `push_true`
压入真值 `True`

### `push_i8 / push_i16 / push_i32 / push_f32`
压入整数或浮点常量

### `push_string`
压入脚本内联字符串

**注意**：
- 它不只是文本内容，还隐含“该字符串在脚本中的地址身份”
- 某些 syscall 会依赖这种“常量字符串地址”语义

### `push_global idx`
压入全局变量 `G[idx]`

### `push_stack idx`
压入当前函数帧中的参数或局部变量

### `push_top`
复制当前栈顶值

### `push_return`
把最近一次 `call/syscall` 的返回值压栈，然后清空返回寄存区

---

## 2.3 表操作类

### `push_global_table idx`
**行为**：
- 弹/读当前栈顶作为 key
- 取 `GT[idx][key]`
- 结果放回当前栈顶位置

### `push_local_table idx`
**行为**：
- 弹/读当前栈顶作为 key
- 取 `LT[idx][key]`
- 结果放回当前栈顶位置

### `pop_global_table idx`
**行为**：
- 栈顶两个值：`key` 与 `value`
- 执行 `GT[idx][key] = value`

### `pop_local_table idx`
**行为**：
- 栈顶两个值：`key` 与 `value`
- 执行 `LT[idx][key] = value`

---

## 2.4 赋值类

### `pop_global idx`
**行为**：
- 弹出栈顶值
- 写入全局变量 `G[idx]`

### `pop_stack idx`
**行为**：
- 弹出栈顶值
- 写入局部变量/参数槽位

---

## 2.5 算术与逻辑类

### `neg`
对栈顶数值取负

### `add/sub/mul/div/mod`
弹两个值做二元运算，并把结果压回

### `bit_test`
对两个栈值执行位测试

**近似效果**：
```lua
S0 = (S0 & S1) ~= 0
```

### `and/or`
不是 Lua 全语义布尔，而更接近：
- 非 `Nil` 视为真
- `Nil` 视为假

### `set_e / set_ne / set_g / set_ge / set_l / set_le`
比较栈顶两个值，把结果转成 `True/Nil`

---

## 3. 典型“单个指令块”模式

下面按“功能块”而不是单条指令来解释 HCB 常见语义。

---

## 3.1 文本打印块

### 典型模式
```text
push_i32 <text_id>
push_string "......"
syscall TextPrint
```

### 含义
- 选择一个文本缓冲槽（text buffer）
- 把字符串压栈
- 调用 `TextPrint`

### 实际效果
通常不只是“立刻显示文本”，而可能包括：
- 向 TextManager 追加文本
- 启动逐字显示/reveal
- 进入 `text_wait`
- 当前脚本线程让出执行，等待文本完成或玩家推进

### Lua-like 近似
```lua
S0 = 0
S1 = "今天的风很大。"
__ret = __syscall("TextPrint", S0, S1)
```

### 说明
这是视觉小说脚本中最常见、最核心的基本块之一。

---

## 3.2 文本样式设置块

### 典型模式
```text
push_i32 <text_id>
push_i32 <font_or_color_1>
push_i32 <font_or_color_2>
...
syscall TextFont / TextColor / TextSize / TextPos / TextSpeed
```

### 含义
设置当前文本缓冲槽的：
- 字体
- 颜色
- 尺寸
- 显示位置
- 显示速度

### 作用
此类块一般不是直接“推进剧情”，而是修改 TextManager 的状态，为后续 `TextPrint` 做准备。

---

## 3.3 资源加载块（图像）

### 典型模式
```text
push_i32 <graph_id>
push_string "graph/xxx"
syscall GraphLoad
```

### 含义
把某个图像资源加载到指定 graph 槽位

### 作用
后续 sprite/prim 往往只引用 graph id，而不是再次引用文件名。

### Lua-like 近似
```lua
S0 = 5
S1 = "graph/title_logo"
__ret = __syscall("GraphLoad", S0, S1)
```

---

## 3.4 图元绑定块（PrimSetSprt / PrimSetXY / PrimSetWH / PrimSetUV）

### 典型模式
```text
push_i32 <prim_id>
push_i32 <graph_id>
push_i32 <u>
push_i32 <v>
syscall PrimSetSprt
...
syscall PrimSetXY
...
syscall PrimSetWH
...
syscall PrimSetUV
```

### 含义
把 graph 贴图加载到某个 prim 上，并设置：
- 位置 XY
- 显示尺寸 WH
- 纹理采样矩形 UV
- Z 顺序 / alpha / blend / draw 等属性

### 实际意义
这是 FVP 画面层最典型的一类“演出块”：

> 脚本并不直接画图，而是通过 syscall 配置 prim，之后由渲染系统绘制。

---

## 3.5 音频播放块

### 典型模式（BGM）
```text
push_i32 <channel>
push_string "bgm/001"
syscall AudioLoad
push_i32 <channel>
push_true
syscall AudioPlay
```

### 含义
- 先把音频文件加载到指定频道
- 再播放，可带 loop 标志

### 典型模式（SE）
```text
push_i32 <slot>
push_string "se/xxx"
syscall SoundLoad
push_i32 <slot>
push_true or push_nil
...
syscall SoundPlay
```

### 作用
这些块通常不阻塞脚本推进，除非后面显式调用等待型指令/系统调用。

---

## 3.6 线程启动块

### 典型模式
```text
push_i32 <thread_id>
push_i32 <function_addr>
syscall ThreadStart
```

### 含义
在指定线程槽位启动一个新的脚本 context

### 特殊点
- 第二个 `push_i32` 不是普通整数，而是**函数地址立即数**
- 回编译时必须参与地址重定位

### Lua-like 近似
```lua
S0 = 3
S1 = 0x00123456
__ret = __syscall("ThreadStart", S0, S1)
```

### 作用
它常用于：
- 并行演出
- 子流程控制
- 独立计时/动画逻辑

---

## 3.7 等待块

### 3.7.1 时间等待
```text
push_i32 <ms>
syscall ThreadWait
```

**作用**：
- 当前线程进入 WAIT 状态
- 直到计时结束再恢复

### 3.7.2 睡眠块
```text
push_i32 <token_or_ms>
syscall ThreadSleep
```

**作用**：
- 当前线程进入 SLEEP 状态
- 等待 `ThreadRaise` 或其他机制恢复

### 3.7.3 文本等待块
通常不是脚本显式写出，而是：

```text
syscall TextPrint
```

内部触发：
- `thread_text_wait`
- `should_break`

### 3.7.4 转场等待块
```text
push_xxx ...
syscall DissolveWait
```

**作用**：
- 若有 dissolve 在进行，则线程挂起直到转场结束

---

## 3.8 条件分支块

### 典型模式
```text
push_stack <a>
push_i32 0
set_e
jz <else_or_end>
```

### 含义
- 先构造一个布尔结果在栈顶
- `jz` 取这个结果，若假则跳转

### Lua-like 近似
```lua
S0 = a0
S1 = 0
S0 = (S0 == S1)
if __is_nil(S0) then goto BB_else end
```

### 说明
在 HCB 中，`if` 语句通常不会直接作为高级结构出现，而是：

> 比较/布尔运算块 + `jz` + fallthrough / `jmp`

组合成的。

---

## 3.9 循环块

### 典型形态
```text
loop_header:
  ...
  <construct condition>
  jz loop_end
  ...
  jmp loop_header
loop_end:
```

### 含义
这是最典型的 while 风格控制流骨架。

### CFG 识别特征
- 有一个后向边（back-edge）
- 跳回起点的 block 会把目标 block 标成 `loop_header`

---

## 3.10 表读写块

### 典型读
```text
push_i32 <key>
push_global_table <table_slot>
```
或
```text
push_i32 <key>
push_local_table <local_idx>
```

### 典型写
```text
push_i32 <key>
push_xxx <value>
pop_global_table <table_slot>
```

### 作用
这类块通常承担：
- 动态数组/映射访问
- 事件结构、返回结构、临时表数据
- 某些 syscall 的复杂参数包装

---

## 4. 如何从块看剧情逻辑

若你将来要“完全解析 HCB 每段具体做了什么”，建议按下面顺序读块：

1. **先看 syscall 名字**
   - Text / Graph / Audio / Movie / Thread / Save / Motion
2. **再看它前面 2~8 条 push 指令准备了什么参数**
3. **再看它是否紧跟 `jz/jmp/ret`**
4. **再结合 CFG 看它属于：**
   - 顺序剧情块
   - 条件分支块
   - 循环块
   - 子函数块
   - 并行线程块

这样往往比死盯单条 opcode 更容易理解整段 HCB 的真实作用。

## 5. 当前脚本输出与本文档的关系

### `hcb_to_ir.py` 输出的 `.ir.json`
- 最适合做精确语义与回编译

### `.cfg.json`
- 最适合看“块与块之间怎么跳”

### `.lua`
- 最适合按本文档中的“功能块”思路阅读

推荐做法：

- 用 `.cfg.json` 找 block
- 用 `.lua` 看 block 做了什么
- 用 `syscall语义数据库` 看 syscall 具体副作用
- 必要时回到 `.ir.json` 看原始参数与地址

## 6. 当前仍需注意的歧义

### 6.1 布尔与整数
不要把 `0/1` 直接当作 Lua/C 风格布尔理解；当前项目规范采用：
- `Nil` 假
- 非 `Nil` 真

### 6.2 `0x25/0x27` 命名
本文档统一采用：
- `0x25 = set_ge`
- `0x27 = set_le`

### 6.3 `push_string`
它不只是“文本常量”，还带有脚本内地址身份。

### 6.4 旧版兼容 syscall
来自 `legacy.rs` 的一些 syscall 适用于旧作/兼容路径，其行为不能直接外推到所有新作。

---

## 7. 一句话总结

如果把 HCB 的分析方法压缩成一句话：

> **先用 opcode 看“值怎样在栈上流动”，再用 syscall 看“这些值怎样改变 GameData”，最后用 CFG 看“这些动作在什么控制流条件下发生”。**

这正是阶段三这套可逆转换脚本存在的意义。
